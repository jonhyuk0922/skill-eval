# skill-eval — 내 스킬, 다섯 축으로 검수

내가 만든 스킬을 **점수 매기려고** 보는 게 아니라, **다음에 뭘 한 가지 고칠지**를 집어주려고 본다.
다섯 축으로 비교하면 어디가 짧은지가 한눈에 보이고, 가장 짧은 축이 곧 오늘 고칠 자리다.

## 언제 쓰나

- 방금 만든(혹은 예전에 만든) 스킬이 쓸 만한지 점검할 때
- 동료 스킬을 리뷰하거나, 마켓에 올리기 전 자가검수할 때
- "이 스킬 어디부터 고치지?"가 막막할 때

## 입력

대상 스킬의 경로 하나.
- 스킬 폴더(`.../skill-name/`) 또는 그 안의 `SKILL.md` 파일 경로
- 경로를 안 주면 **한 번만** 묻고 진행한다. 추측해서 임의 경로를 평가하지 않는다.

## 절차

1. **대상 모으기** — `scripts/collect.py <경로>` 를 실행한다.
   SKILL.md 본문 + 보조 파일(references·scripts) 목록 + 휴리스틱 힌트 + 빈 채점표가 한 번에 나온다.
   (스크립트를 못 쓰는 환경이면 `Read`로 SKILL.md와 보조 파일을 직접 읽는다.)
2. **다섯 축 채점** — `references/rubric.md` 의 "여기까지면 합격" 줄을 잣대로 각 축을 매긴다.
   `합격(2) / 부분(1) / 미달(0)`. **판정마다 본문에서 근거 한 줄을 인용**한다. 근거 없는 점수는 금지.
   - ②(검증 루프)·③(진짜 업무성)은 collect.py 휴리스틱이 약하다 — 힌트가 조용해도 본문에 *실제 입력→출력 예시*와 *구체적 반복 업무*가 있는지 눈으로 직접 확인한다.
3. **가장 짧은 축** — 점수가 가장 낮은 축이 오늘 고칠 자리. (동점이면 ①→②→③ 순서로 앞 축 우선.)
4. **3대 결함 탐지** — `references/rubric.md` 의 결함 표에서 해당되는 것을 고른다:
   뭉뚱그린 명세 / 한 방에 끝(검증 루프 없음) / toy 예제.
5. **한 줄 처방** — 가장 짧은 축을 한 단계 올리는, **복붙해서 바로 쓰는 프롬프트 한 줄**. 추상어("개선하라")는 금지.
6. **출력** — `references/scorecard-template.md` 형식으로 채점표를 낸다.

## 출력 형식

```
## 검수 결과 — <스킬 이름>   합계 N/10

| 축 | 판정 | 근거 (본문 한 줄) |
|---|---|---|
| ① 명세 또렷함 | 합격/부분/미달 | "…" |
| ② 검증 루프 | … | "…" |
| ③ 진짜 업무성 | … | "…" |
| ④ 동료 이식성 | … | "…" |
| ⑤ 임팩트 | … | "…" |

🎯 오늘 고칠 자리: <가장 짧은 축>
🩺 걸린 결함: <뭉뚱그린 명세 / 한 방에 끝 / toy 예제 중>
💊 한 줄 처방: <복붙용 프롬프트 한 줄>
```

## 규칙

- **목적은 점수가 아니라 다음 한 걸음.** 합계보다 "가장 짧은 축 + 처방"이 핵심이다.
- 못 만들어 온 스킬도 평가 가능 — 빈 칸이 곧 "다음엔 이렇게"라는 설계도다.
- 처방은 한 줄, 복붙 가능, 대상 스킬에 바로 끼울 수 있게 구체적으로.
- 다섯 축을 한꺼번에 다 고치라고 하지 않는다. **한 번에 한 축.**
- 사용자가 원하면 한 줄 처방을 **바로 적용**해, 가장 짧은 축을 끌어올린 고친 SKILL.md를 보여준다 (점검 → 수정까지 한 흐름).
